## 六、注意事项

1. 不要使用包含真实隐私或敏感信息的资料。
2. 如果使用联网模型，应避免上传不可公开资料。
3. 原神主题资料可以使用教学模拟文本；报告中应说明资料是否代表实时版本信息。
4. 权限控制必须发生在检索阶段，不能只依赖提示词要求模型不泄露。
5. 如果系统检测到资料冲突，不应强行给出唯一结论，应展示冲突来源并说明处理方式。
6. 代码应能在普通同学电脑上运行；如 ChromaDB 或 Qwen 接入失败，应提交错误截图、已完成路线的运行结果和原因分析。
7. 学号不得只写在报告封面中；提交源码、运行日志或关键截图应能证明代码中包含本人学号。

## 七、常见问题

### Q1: 检索命中率低是否一定扣很多分？

不一定。只要报告能清楚分析原因并提出改进方案，仍可体现实验价值。

### Q2: 可以使用自己的原神主题资料吗？

可以，但资料内容应适合公开展示，不包含个人隐私、账号信息、抽卡记录截图或其他敏感信息。

### Q3: 为什么评分中要求引用来源？

因为 RAG 系统的可信度不只来自“回答看起来合理”，还来自回答能否追溯到具体资料片段。没有引用来源，就很难判断答案是否真的基于资料库。

### Q4: 权限过滤截图应该体现什么？

同一个问题分别用 `student` 和 `teacher` 角色检索。报告中应能看到 `student` 结果不包含教师私有资料，而 `teacher` 结果可以包含教师资料。

### Q5: ChromaDB 截图应该体现什么？

至少体现两点：`chroma_records` 写入数量，以及一次通过 ChromaDB 查询得到的回答、引用来源、角色过滤或过期过滤结果。

### Q6: 是否必须同时完成两个 Qwen 模型？

不是。至少完成一个本地 Qwen 模型接入即可。若硬件条件允许，可以同时运行 `qwen3.6:35b-a3b` 与 `qwen3.6:27b`，比较两个模型在引用保持、回答稳定性和响应速度上的差异。
