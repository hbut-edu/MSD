## 三、报告内容要求

### 3.1 实验目的与系统定位

报告应说明什么是多智能体系统，什么时候需要拆成多个 Agent，什么时候不应该为了形式而拆分。报告还应说明本章为什么位于“Agent 能调用模型和工具”与“Agent 系统可运维”之间，并解释以下对象的职责：

- Provider：模型服务来源和鉴权配置。
- Agent：绑定模型、系统提示词和任务角色。
- Channel：外部消息进入网关的通道。
- Router：根据规则分发消息；LLM Router 则根据自然语言意图输出结构化编排计划，决定多个 Agent 的参与顺序、复核点和最终汇总者。
- Skill/MCP：后续能力扩展和工具协议的入口；本章要求为四个入口创建轻量角色 Skill，用于标清职责边界。
- 配置文件和日志：用于复现、迁移、排错和审计。

### 3.2 环境与版本说明

报告应展示并解释：

- OpenClaw 版本。
- Hermes 版本。
- Node 或 Python 版本。
- Ollama 版本和本地模型名称。
- 使用的云端模型服务类型，例如 DeepSeek OpenAI 兼容接口、Coding Plan、在线推理接口或课堂统一模型服务。
- 云端模型成本方案：若使用 DeepSeek，应说明采用官方 `deepseek-v4-flash` 低价路线、`deepseek-v4-pro` 少量高质量路线，还是 OpenRouter 免费试跑路线；报告应记录 Base URL、模型名、预算或余额控制方式、一次调用的 token 用量摘要和查询价格日期。
- OpenClaw 实际配置文件路径。
- Hermes 配置目录、数据目录或等价配置观察位置。
- 如果手册命令与本机命令不同，应说明版本差异和实际使用的等价命令。
- 对每个配置阶段，应说明采用飞书后台、CLI、`.env` 或结构化配置文件的原因，避免把同一项配置重复堆叠成命令行和配置文件两套互相冲突的路线。

### 3.3 网关启动与健康检查

报告应包含：

- 网关启动命令。
- 健康检查返回结果或等价状态检查。
- 监听地址、端口或 daemon 状态。
- 防火墙、端口占用、权限问题等排错记录。
- 一段脱敏后的启动日志或关键截图。

### 3.4 云端 Provider 与 Agent 接入

报告应说明：

- Provider 名称、Base URL 类型和鉴权方式。
- Agent 名称、模型名或 Endpoint、温度和系统提示词用途。
- 为什么 Provider 与 Agent 应拆分。
- DeepSeek/OpenAI 兼容接口、Coding Plan 与在线推理接口的区别，以及错误 Base URL 或模型别名可能带来的费用和可用性风险。
- DeepSeek 低成本实践方案：默认推荐 `deepseek-v4-flash`；`deepseek-v4-pro` 只用于少量复杂规划或最终汇总；没有余额时可用 OpenRouter `openrouter/free` 或当日可用 DeepSeek `:free` 模型做连通性试跑，但需说明免费模型的请求限制、可用性波动和实际返回模型。
- 至少一次云端 Agent 调用验证结果。

### 3.5 飞书 Channel/Gateway 与最小权限

报告应说明：

- OpenClaw 与 Hermes 使用的飞书企业自建应用创建过程。标准做法是四个测试机器人分别服务 `OC-DeepSeek-<STUDENT_ID>`、`OC-Qwen-<STUDENT_ID>`、`Hermes-DeepSeek-<STUDENT_ID>`、`Hermes-Qwen-<STUDENT_ID>`；其中 `<STUDENT_ID>` 为本人完整学号。若账号限制导致无法创建四个机器人，必须经授课老师确认替代方案。
- App ID 与 App Secret 的保存和脱敏方式。
- WebSocket 或长连接事件接收配置。
- 已申请权限及其必要性。
- 测试群 `chat_id` 与私聊用户 `open_id` 的来源和使用方式：`chat_id` 可从飞书消息事件、发送消息回执或 Gateway 日志中获取；`open_id` 应分别从四个机器人各自收到的私聊事件、pairing 提示或 Gateway 日志中获取；OpenClaw 应说明 `dmPolicy/allowFrom`、`groupPolicy/groups` 白名单，以及本机版本是否还需要 `groupAllowFrom` 兼容字段；Hermes 应说明 `FEISHU_ALLOWED_USERS`、`FEISHU_HOME_CHANNEL` 或 `group_rules`。
- 权限设定过程：OpenClaw 应说明如何从日志或会话中取得两个机器人各自的 `open_id`，如何把私聊和测试群分别收敛到 allowlist；Hermes 应说明 pairing code 如何审批，或如何将用户写入 `FEISHU_ALLOWED_USERS`，并说明测试群外默认不响应。
- 测试消息是否分别抵达 OpenClaw Channel 与 Hermes Gateway。
- 如果能收到消息但不能回复，排查顺序是什么。
- OpenClaw 和 Hermes 是否能同时服务：四机器人模式下应并行运行；只有共用同一个飞书机器人时，才需要停止其中一个 Gateway 或配置额外过滤，以避免双响应、日志混淆和权限边界不清。

### 3.6 本地 Ollama Agent 与对比分析

报告应说明：

- 本地 Ollama Provider 配置。
- 本地 Agent 使用的模型、提示词和验证命令，推荐记录 `qwen3.6:27b` 或课堂实际模型。
- 本地模型与云端模型在性能、隐私、成本、稳定性和硬件依赖上的差异。
- 是否进行断网或半离线验证，以及观察到的现象。

### 3.7 Router、LLM Router 与端到端验收

报告应展示：

- `/code` 路由到云端代码 Agent 的验证结果。
- `/chat` 路由到本地聊天 Agent 的验证结果。
- 一次复合指令的 LLM Router 结构化输出，说明意图识别、Agent 编排顺序、安全复核和最终汇总者。
- 至少一次低置信度 LLM Router 案例，说明系统是否选择反问或人工确认，而不是强行路由。
- 一次“实验4作业初筛与风险检查助手”案例回放，说明四个智能体分别承担什么角色、使用什么 Skill、如何交接结果，以及最终生成了什么报告。
- `notes/agentops_case_delivery.md` 中应记录四个入口的调用命令、脱敏输出摘要、耗时、no-thinking 检查结果和最终报告结构。
- 飞书客户端回复、网关日志和 Agent 调用结果三类证据。
- 至少一次未知前缀、错误路由或失败请求的观察。
- 路由误配或 LLM Router 误判后可能产生的用户体验、安全、权限或成本问题。

### 3.8 Hermes 双 Agent 入口与飞书 Gateway

报告应说明：

- Hermes 安装方式、版本检查和 `hermes doctor` 结果。
- `hermes-exp4-deepseek` 入口的 Provider、Base URL 类型、模型名和 CLI 验证结果。
- `hermes-exp4-qwen` 入口的本地 Ollama Custom provider、模型名和 CLI 验证结果。
- Hermes 飞书 Gateway 从零配置过程、状态检查和消息抵达证据。
- 两个 Hermes Agent 入口在飞书中的验收方式：标准做法是 `Hermes-DeepSeek-<STUDENT_ID>` 与 `Hermes-Qwen-<STUDENT_ID>` 两个机器人并行，对应两个 Profile/Gateway；分时切换默认模型只能作为授课老师确认后的替代方案。
- Hermes Gateway 中的编排观察：`Hermes-DeepSeek-<STUDENT_ID>` 可作为规划/Router/汇总入口，`Hermes-Qwen-<STUDENT_ID>` 可作为本地安全复核入口；报告应说明一次复合指令如何在两个 profile 间拆分与回收。
- 飞书群标准验收必须分别直接 `@` 具体机器人；`@全员` 或 `@所有人` 默认不作为四机器人均已接入的有效证据。若报告使用 `@所有人` 作为附加验证，必须额外说明飞书“群组中所有消息”权限、OpenClaw `respondToMentionAll`、Hermes `@_all` 处理、测试群白名单和费用/误触发风险。
- Hermes 工具配置、安全审批、记忆、Skill 或 MCP 入口的观察结果。
- 当前课堂环境中哪些 Hermes 能力适合启用，哪些不适合启用，原因是什么。

### 3.9 配置落盘、双框架对比与风险分析

报告应结合脱敏配置片段说明：

- OpenClaw 中 Provider、Agent、Channel、Router 分别出现在配置文件的什么位置。
- Hermes 中两个 Agent 入口、飞书 Gateway、记忆或 Skill 配置分别在哪里观察。
- 哪些配置可以迁移，哪些配置必须重新申请或重新生成。
- 配置文件与界面/向导之间是否存在不一致。
- 日志中哪些字段可用于排错，哪些字段需要脱敏。
- OpenClaw 与 Hermes 在安装体验、模型配置、消息通道、规则 Router、LLM Router、工作流编排、Skill/记忆、安全控制、运维复杂度和课堂适配性上的优缺点。
- 至少 4 类 AgentOps 风险及其应对措施。

## 五、加分项

在完成基础要求后，可以选择以下扩展，最高加 10 分：

- 编写一键配置或检查脚本，能自动检查版本、配置路径、Provider、Agent 和 Router：+3 分
- 提供可复用的脱敏配置模板，并说明哪些字段需要通过环境变量注入：+3 分
- 增加简单可观测性统计，例如路由命中次数、请求耗时、失败原因分类：+3 分
- 在授课老师确认下额外接入钉钉、企业微信、本地 Webhook 或其他 Channel，并完成与飞书通道的对比：+3 分
- 在授课老师确认下接入第三个云端模型或第二个本地模型，并完成对比分析：+3 分
- 设计凭证轮换、权限收敛、日志脱敏或预算告警方案，并说明如何落地：+4 分

加分项不能替代基础链路。若 OpenClaw 主链路或 Hermes 双 Agent 飞书 Gateway 验收未完成，加分项不计入总评。

## 六、注意事项

1. API Key、App Secret、Hermes Gateway 凭证、Token、Cookie、个人账号隐私或企业内部敏感信息的真实值来自对应控制台、`.env` 或网关日志；提交材料中的共享片段使用占位符说明来源和用途。
2. 不得使用他人账号、他人企业应用或未经授权的群聊进行实验。
3. 云端模型调用前应确认费用规则，不要把课堂套餐接口和个人按量接口混用。
4. 如果某一步没有完全跑通，应如实说明卡点、已尝试方案、日志证据和下一步判断，不得伪造截图或输出。
5. 截图应能看清关键命令、返回结果和配置对象；如果页面上出现敏感字段，应只保留能说明来源、状态和路径的部分。
6. 飞书应用列表、发布成功页、群机器人列表和机器人对话输出属于推荐截图证据；截图应服务于说明“从零创建 -> 发布 -> 加入群聊 -> 能够对话”的链路，不应只堆放界面图片。
7. 配置文件如需提交，使用占位符样例说明字段来源、放置位置和验证方式；Hermes 记忆、Skill 和会话记录也应先检查是否包含隐私。
8. 飞书部分必须体现 OpenClaw 与 Hermes 两侧的“创建应用 -> 配置权限 -> 接入 Gateway -> 验证消息”的过程；只提交已有机器人成功截图但没有配置与验证证据，不视为完整完成。
9. 同学之间可以讨论排错思路，但报告、配置记录、日志证据和反思必须体现个人实验过程。

## 七、常见问题

### Q1: 我的 OpenClaw 命令和手册不一样，会扣分吗？

不会。OpenClaw 版本迭代较快，评分重点是是否能完成等价的网关、Provider、Agent、Channel、Router 和配置复盘链路。报告中应写清楚本机版本、实际命令和差异原因。

### Q2: Hermes 必须完整接入飞书吗？

必须。本章基础要求是 OpenClaw 与 Hermes 都接入飞书。Hermes 还必须配置 `hermes-exp4-deepseek` 和 `hermes-exp4-qwen` 两个 Agent 入口，并分别给出 CLI 验证和飞书 Gateway 验证。标准做法是四个飞书测试机器人分别对应两个 OpenClaw Agent 和两个 Hermes profile；若因账号限制无法做到，必须说明如何通过分时启停避免双响应，并取得授课老师确认。

### Q3: Hermes 在 Windows 上安装失败怎么办？

优先使用 WSL2 完成课堂验收，并记录失败现象、系统版本、实际命令和 `hermes doctor` 或安装日志。若使用 Windows 原生安装，也应说明哪些能力可用、哪些能力受限。

### Q4: 没有飞书企业账号怎么办？

先向授课老师确认替代方案。可以使用本文档指定的测试企业、模拟 Channel、本地 Webhook 或其他企业 IM。替代方案仍需保留最小权限、消息入口、路由命中和日志复盘这几个核心观察点。

### Q5: 云端模型一直失败，是否可以只做本地模型？

基础评分要求包含云端 Provider 与本地 Provider 的对比。若云端账号或额度受限，应提交错误日志、配置检查记录和授课老师确认说明；只做本地模型通常不能获得云端 Provider 与多模型对比部分的完整分数。

### Q6: 日志需要提交完整文件吗？

不需要提交很长的完整日志。建议截取能证明启动、Channel 连接、路由命中、模型调用和失败案例的关键片段，并用占位符替换凭证、用户标识和群聊标识。

### Q7: 为什么要求失败案例？

AgentOps 不是只关注“跑通一次”，还关注系统如何排错、恢复和审计。一次真实或有意构造的失败案例能帮助同学们理解配置、路由、权限和日志之间的关系。
