# 5. Skill 工程实战：智能体能力扩展与工作流沉淀 - 实验手册

## 实验主题

本实验围绕“如何通过 Skill 增强智能体能力”展开。前序实验已经让同学们体验了 Agent、MCP、RAG、多模型网关和路由配置，本实验进一步讨论：当一个 Agent 工作流已经跑通之后，如何把其中稳定、可复用、可解释的步骤沉淀成 Skill，让智能体在后续任务中更稳定地复用这套方法。

本实验不把 Skill 当作一个神秘插件，而是先从同学们熟悉的问题进入：如果每次都要让 AI 从米游社、游戏官网、HoYoLAB 或课堂指定游戏资讯站中抓取并整理“原神激活码/兑换码信息”，都需要反复提醒它“只抓取公开页面、不要编造最新兑换码、记录来源链接、抽取生效时间和失效时间、提醒防诈骗”，不仅浪费上下文，也容易遗漏关键安全边界。Skill 的作用，就是把这些反复出现的任务经验整理成一份面向 Agent 的能力说明，让 Agent 在合适任务中自动加载并遵循。

本实验的主线是：

```text
一次成功任务 -> 工作流记录 -> 手写最小 Skill -> 挂载到 Agent -> 复现实验
             -> AI 辅助总结 Skill -> 人工校订 -> 有无 Skill 对比 -> 外部 Skill 复用
```

## 前后课程关系

本实验插入在第 4 章和第 5 章之间，用来承接和过渡：

- 第 4 章已经让同学们看到 AgentOps 网关中的 Provider、Agent、Channel、Router、Skill/MCP 等对象。
- 本实验专门讲清楚 Skill：它是什么、由哪些部分组成、如何编写、如何挂载、如何从工作流中沉淀。
- 第 5 章会继续讨论提示词工程验证与评估，本实验中的“有无 Skill 对比”正好为后续评估实验提供直观样本。
- 第 6 章会深入 Harness 与安全护栏，因此本实验只讲 Skill 的边界声明和失败处理，不展开工具白名单、路径隔离等代码级安全实现。

可以把本实验理解为从“会搭 Agent 系统”走向“会沉淀 Agent 能力”的关键一环。

## 实验目标

完成本实验后，同学们应能够：

1. 理解 Skill 是什么，以及它与 Prompt、System Prompt、Tool、MCP、Plugin 的区别。
2. 说明 Skill 对智能体稳定性、复用性、上下文效率和团队协作的作用。
3. 拆解一个 Skill 的核心组成部分，包括元信息、触发条件、适用边界、工作流程、工具规范、输出契约、校验步骤、失败处理和资源文件。
4. 手动编写一个最小可用 `SKILL.md`。
5. 将“原神官方激活码/兑换码信息获取”Skill 挂载到本地 Agent 或课程实验 Agent 的能力目录，并验证 Agent 是否按 Skill 工作。
6. 在跑通一个真实工作流后，使用 AI 辅助总结 Skill，并进行人工校订。
7. 设计有无 Skill 的 A/B 对比实验，观察工作成果、完成效率、对话轮次和 Token 消耗差异。
8. 了解可以直接复用或改造的外部 Skill 类型，并能判断何时应该直接使用、局部改造或重新编写。

## 实验安全注意事项

1. Skill 中不要写入真实 API Key、Token、App Secret、账号密码或企业内部数据。
2. Skill 可以描述工具使用流程，但不能替代代码级权限控制；涉及文件、网络、数据库或系统命令时，后续仍需要 Harness 或等价安全层。
3. 本实验中的 Token 消耗统计以观察和对比为主，不追求绝对精确。若平台提供用量统计，应优先使用平台统计。
4. 使用外部 Skill 前，应先阅读其 `SKILL.md`，确认它的适用范围、依赖工具和潜在风险。
5. AI 辅助总结 Skill 后必须人工校订，删除敏感信息、过度泛化内容和不可靠步骤。
6. 实验报告中的对话记录和截图应脱敏，不要暴露本机绝对路径中的个人隐私信息。
7. 本实验中的“原神激活码”统一按官方兑换码、直播兑换码、活动礼包码等公开信息处理，只允许抓取公开页面中的公开信息，并进行来源核验、时间解析、过期判断和防诈骗提醒。
8. 抓取米游社、游戏官网、游戏资讯站等页面时，应遵守网站服务条款、`robots.txt`、访问频率限制和课堂要求；不得绕过登录、验证码、反爬机制或访问非公开接口。
9. 不得编写、传播或尝试使用盗号、撞库、刷码、破解、脚本抢码、灰色交易、虚假抽奖引流等方法。

## 第一阶段：Skill 概念导入

### 目标

理解 Skill 的基本定位：Skill 是写给 Agent 看的能力说明，它将某类任务中的专门知识、操作流程、工具使用方式、输出格式和验收规则组织起来，让 Agent 在合适场景中按稳定方法工作。

### 从一个熟悉问题开始

假设同学们已经完成过一次“原神官方激活码/兑换码信息获取指南”整理任务。每次让 AI 帮忙设计或扩展这类任务时，都需要反复提醒：

- 只使用官方渠道或可信公开渠道，不编造所谓“最新可用码”。
- 区分兑换码、活动码、直播码、礼包码、过期码和诈骗链接。
- 输出时说明信息来源、查询时间和有效性不确定性。
- 对第三方网站、交易群、抽奖链接和要求登录的页面保持警惕。
- 如果需要实时查询，应先核验来源，再给出谨慎结论。

如果这些要求每次都靠人工重新输入，容易出现三个问题：

| 问题 | 表现 | Skill 能提供的帮助 |
| :-- | :-- | :-- |
| 重复提示 | 每次都要重新解释格式和流程 | 把流程沉淀为可复用说明 |
| 输出漂移 | AI 有时编造兑换码，有时写成引流攻略 | 明确官方来源、核验步骤和防诈骗边界 |
| 上下文浪费 | 大量 Token 用于重复说明背景 | 让 Agent 只在需要时加载任务专用知识 |

因此，可以先把 Skill 理解为：**把一类任务的稳定经验写成 Agent 能读懂、能遵循、能复用的能力包**。

### Skill 与相关概念对比

| 概念 | 主要作用 | 是否强调复用工作流 | 典型例子 |
| :-- | :-- | :-- | :-- |
| Prompt | 当前这一次请求的任务说明 | 不一定 | “帮我写一个实验大纲” |
| System Prompt | Agent 的长期角色和全局规则 | 有，但通常较宽泛 | “你是一个编码助手” |
| Tool | 可被调用的具体函数或外部能力 | 不负责完整流程 | 读取文件、搜索网页、运行测试 |
| MCP | 描述工具能力、参数和调用协议 | 偏接口层 | 文件系统 MCP、数据库 MCP |
| Plugin | 打包一组工具、技能或应用集成 | 可能包含多个能力 | GitHub、Gmail、文档处理插件 |
| Skill | 面向特定任务的能力说明和流程封装 | 是 | 原神兑换码抓取 Skill、PPT 制作 Skill、PR 评论处理 Skill |

Skill 不等同于工具。工具更像“手”；Skill 更像“这类任务应该怎么做的操作说明”。Skill 也不等同于安全控制。Skill 可以说明边界，但真正阻止危险行为仍需要 Harness、权限系统或平台约束。

## 第二阶段：Skill 结构拆解

### 目标

掌握一个 Skill 的常见组成部分。不同 Agent 平台的目录约定可能不同，但一个可维护的 Skill 通常都包含元信息、正文说明和可选资源。

### 典型目录结构

一个常见 Skill 可以组织为：

```text
skill-name/
├── SKILL.md                 # 必需：Skill 元信息和核心说明
├── scripts/                 # 可选：可重复执行的脚本
├── references/              # 可选：按需读取的长参考资料
└── assets/                  # 可选：模板、图片、示例文件等输出资源
```

本实验先只要求同学们完成 `SKILL.md`。当任务变复杂时，再考虑增加 `scripts/`、`references/` 和 `assets/`。

### `SKILL.md` 的核心组成

| 组成部分 | 作用 | 编写要点 |
| :-- | :-- | :-- |
| YAML Frontmatter | 让 Agent 发现 Skill，并判断何时触发 | 至少包含 `name` 和 `description` |
| 名称 | 标识 Skill | 使用小写字母、数字和连字符，例如 `genshin-code-guide` |
| 描述 | 说明何时使用 Skill | 要写清楚触发场景，避免过宽或过窄 |
| 适用场景 | 明确哪些任务应该用 | 用具体任务描述，不写空泛口号 |
| 不适用场景 | 防止误触发 | 说明哪些任务应交给其他 Skill 或普通对话 |
| 输入要求 | 告诉 Agent 开始前需要哪些资料 | 例如目标地区、查询时间、可用渠道、输出格式 |
| 工作流程 | 固化稳定步骤 | 按先读资料、再分析、再生成、再校验排序 |
| 工具使用规范 | 说明应优先用哪些工具或命令 | 例如优先检查官方公告、官方社区或课程提供的资料 |
| 输出契约 | 规定最终产物格式 | 例如表格、Markdown、JSON、报告章节 |
| 校验清单 | 帮助 Agent 自查 | 检查是否漏目标、漏安全提示、漏验证步骤 |
| 失败处理 | 遇到缺资料或冲突时怎么做 | 说明何时继续、何时说明假设、何时询问 |
| 示例 | 提供短输入和短输出片段 | 用少量高质量例子，不堆长篇样例 |
| 资源引用 | 指向脚本、参考资料或模板 | 让 Agent 需要时再读取，节省上下文 |

### 渐进披露原则

Skill 不是越长越好。Agent 的上下文窗口是有限资源，Skill 应该只放真正能提升任务表现的信息。

一个实用原则是：

```text
必须每次知道的规则 -> 写入 SKILL.md
只有某些场景需要的长资料 -> 放入 references/
需要稳定执行的重复代码 -> 放入 scripts/
最终产物要复用的模板或素材 -> 放入 assets/
```

如果 `SKILL.md` 写得过长，Agent 每次触发都会加载大量无关内容，反而增加 Token 消耗。

## 第三阶段：手写最小 Skill

### 目标

同学们将围绕一个具体任务手写 Skill：帮助 Agent 从米游社、游戏官网、游戏资讯站等公开来源中抓取原神兑换码信息，并抽取兑换码、生效时间、失效时间、来源链接、抓取时间和可信度标记。这个任务有明确流程、格式和安全边界，适合观察 Skill 是否真的改变 Agent 的工作方式。

### 操作步骤一：确定任务边界

先在 `workflows/manual_run_notes.md` 中写下这个 Skill 要解决的问题：

```markdown
# 手动工作流记录

## 任务名称

原神官方兑换码与活动码信息抓取。

## 为什么需要 Skill

每次抓取原神兑换码信息时，都需要重复说明来源范围、抓取边界、兑换码格式、生效时间、失效时间、来源可信度和防诈骗要求。

## 期望 Agent 做到

- 优先抓取官方或高可信公开来源，例如米游社公开帖子、游戏官网公告、HoYoLAB 公告、官方社交媒体公告和课堂指定游戏资讯站。
- 抓取前检查页面是否公开可访问，遵守访问频率和站点规则。
- 从正文中抽取兑换码、发布时间、生效时间、失效时间、适用服务器和来源链接。
- 对未说明失效时间的信息标记为 `unknown`，不直接声称“永久有效”。
- 对第三方来源进行可信度标注，并提醒同学不要点击要求登录、付费、加群或下载不明文件的链接。
```

### 操作步骤二：编写 `SKILL.md`

将下面内容写入 `skills/genshin-code-guide/SKILL.md`。这份 Skill 是教学版示例，同学们可以根据课堂指定来源继续修改。

```markdown
---
name: genshin-code-guide
description: 当需要从米游社、游戏官网、HoYoLAB、官方社交媒体或课堂指定游戏资讯站等公开来源抓取原神兑换码、直播码、活动礼包码信息，并抽取生效时间、失效时间、来源链接和可信度时使用。只处理公开页面，不绕过登录、验证码或反爬限制。
---

# Genshin Code Guide

## 适用场景

使用本 Skill 处理以下任务：

- 抓取公开页面中的原神兑换码、直播兑换码、活动礼包码信息。
- 从页面标题、正文和发布时间中抽取兑换码、来源和时间信息。
- 判断兑换码是否仍可能有效，并说明判断依据。
- 汇总多个来源，去重并标注官方来源、半官方来源和第三方来源。
- 生成面向用户的安全提示，提醒不要点击可疑链接或提交账号密码。

## 不适用场景

以下任务不使用本 Skill：

- 获取、购买、交换或售卖私人兑换码。
- 绕过登录、验证码、反爬机制或访问非公开接口。
- 自动登录游戏账号、提交兑换、批量试码或刷码。
- 生成看似真实但未经来源核验的“最新可用兑换码”。

## 工作流程

1. 明确抓取范围：优先官方来源，其次课堂指定游戏资讯站；记录每个来源的名称、URL、来源类型和抓取时间。
2. 抓取前检查页面是否公开可访问；不要尝试绕过登录、验证码、付费墙、App 私有接口或反爬限制。
3. 获取页面标题、正文、发布时间和链接；若页面依赖动态渲染，可在课堂允许时使用浏览器自动化获取可见文本。
4. 使用正则和上下文规则抽取疑似兑换码，并过滤明显不是兑换码的订单号、UID、版本号、URL 片段。
5. 从正文附近抽取时间表达，例如“即日起”“今日 12:00 生效”“5 月 16 日 23:59 前”“有效期至”“直播结束后次日中午前”等。
6. 将发布时间、生效时间、失效时间统一成带时区的字段；无法确定时标记为 `unknown`，并保留原文片段。
7. 对同一兑换码按代码值去重，保留最可信来源和所有佐证来源。
8. 给每条记录标注状态：`active_candidate`、`expired`、`unknown`、`unverified`。
9. 输出结果前检查是否包含来源链接、抓取时间、生效/失效时间、可信度和安全提醒。

## 输出要求

- 输出 Markdown 表格和 JSON 两种结构，便于人读和程序继续处理。
- 每条兑换码至少包含 `code`、`source_name`、`source_url`、`source_type`、`published_at`、`valid_from`、`valid_until`、`status`、`confidence`、`captured_at`、`evidence_text`。
- 不把第三方未核验信息写成官方结论。
- 不输出要求登录、付费、加群、下载 App 外安装包或填写账号密码的可疑链接。
- 如果没有找到可核验兑换码，应明确说明“未在允许来源中找到可核验信息”。

## 自查清单

完成后检查：

- 是否只抓取公开页面。
- 是否记录抓取时间和来源 URL。
- 是否抽取并归一化生效时间、失效时间和发布时间。
- 是否把未知时间明确标记为 `unknown`。
- 是否区分官方、半官方和第三方来源。
- 是否对过期或无法验证的兑换码给出谨慎提示。
- 是否避免编造“最新可用码”。
```

### 操作步骤三：检查 Skill 是否过度泛化

写完后，同学们需要回答三个问题：

1. 这个 Skill 触发场景是否足够明确？
2. 如果 Agent 每次触发都读取这份 Skill，是否会浪费大量无关上下文？
3. 这份 Skill 是否把“建议”误写成了“安全保证”？

如果答案不理想，应先修改 `description`、适用场景和工作流程，再进入挂载验证。

## 第四阶段：挂载到 Agent 并验证

### 目标

让 Agent 能发现并使用刚才编写的 Skill。不同平台的挂载方式会不同，本实验给出两种常见思路：本地 Skill 目录挂载和 Agent 配置文件挂载。课堂上以任课教师或助教提供的实际 Agent 环境为准。

### 方式一：挂载到本地 Skill 目录

如果使用的 Agent 支持从固定 Skill 目录自动发现能力，可以将整个 Skill 文件夹复制到指定目录。以 Codex 类环境为例，通常会使用 `CODEX_HOME/skills` 或 `~/.codex/skills`。

下面命令用于把当前实验 Skill 复制到本地 Skill 目录。执行前请确认路径符合本机 Agent 的实际约定。

```bash
# 如果 CODEX_HOME 未设置，可以将其理解为 ~/.codex。
mkdir -p "${CODEX_HOME:-$HOME/.codex}/skills/genshin-code-guide"

cp skills/genshin-code-guide/SKILL.md \
  "${CODEX_HOME:-$HOME/.codex}/skills/genshin-code-guide/SKILL.md"
```

验证方法：

1. 重启或刷新 Agent 会话。
2. 提出一个明确会触发该 Skill 的任务，例如“请从课堂指定来源抓取原神兑换码信息，并标出生效时间和失效时间”。
3. 观察 Agent 是否先确认来源范围、公开访问边界、抓取时间字段和有效期字段。
4. 如果 Agent 没有触发，优先检查 `description` 是否写清楚了使用场景。

### 方式二：通过 Agent 配置挂载

如果课程使用自定义 Agent 框架，可以在 Agent 配置中显式声明 Skill 路径。下面是一个教学用伪配置，具体字段名以课堂框架为准。

```yaml
agent:
  name: genshin-code-agent
  model: qwen-or-other-model
  skills:
    - path: ./skills/genshin-code-guide
```

验证时，不要求同学们证明 Agent “记住”了全部 Skill 内容，而是观察它是否按 Skill 的关键流程行动：

| 验证点 | 通过表现 |
| :-- | :-- |
| 是否确认来源范围 | 优先官方来源，并说明第三方来源需要核验 |
| 是否遵守抓取边界 | 不要求登录、不绕过验证码、不访问非公开接口 |
| 是否抽取时间字段 | 记录发布时间、生效时间、失效时间和抓取时间 |
| 是否处理未知时间 | 对无法确定的字段标记 `unknown`，不编造时间 |
| 是否自查 | 输出前检查来源、时间、状态、可信度和安全提醒 |

### 常见问题

如果 Skill 未被触发，常见原因包括：

- `SKILL.md` 不在 Agent 可发现目录中。
- `name` 使用了空格、中文或特殊符号，平台无法识别。
- `description` 太短，只写“课程助手”，Agent 无法判断何时使用。
- 任务描述过于宽泛，没有命中 Skill 场景。
- 当前 Agent 平台不支持自动 Skill 发现，需要显式配置。

## 第八阶段：工程化维护总结

### 目标

理解 Skill 不是一次性文档，而是团队知识资产。一个有用的 Skill 应该能被版本管理、被复查、被验证、被迭代。

### 维护建议

| 维护事项 | 建议做法 |
| :-- | :-- |
| 版本管理 | 将 Skill、抓取脚本、来源清单和样例输出一起纳入 Git |
| 命名规范 | 使用小写英文、数字和连字符 |
| 触发描述 | 每次修改后检查 `description` 是否仍准确 |
| 示例任务 | 保留 1-2 个最小验证任务 |
| 敏感信息 | 永远不把真实密钥、账号、内部数据写入 Skill |
| 团队共享 | 让使用者知道适用场景和不适用场景 |
| 失效处理 | 当平台、命令或目录结构变化时及时更新 |
| 质量评估 | 定期做有无 Skill 对比或版本间对比 |

### 阶段小结

本实验完成后，同学们应能形成一个重要工程判断：Prompt 更适合表达当前这一次任务，Skill 更适合沉淀一类任务的稳定能力。一个成熟 Agent 系统不只依赖“更会说的提示词”，还需要把可复用经验、工具使用方法、输出契约和校验规则组织起来，让智能体的能力可以被复现、被改进、被团队共享。

## 故障排除 FAQ

### Q1: Skill 和普通提示词最大的区别是什么？

**A:** 普通提示词通常只服务当前一次请求；Skill 面向一类重复任务，强调可复用的工作流程、输入要求、输出契约和校验规则。

### Q2: Skill 写得越详细越好吗？

**A:** 不是。Skill 会占用上下文。应把每次都必须知道的内容放进 `SKILL.md`，把长资料放到 `references/`，把重复脚本放到 `scripts/`。

### Q3: Agent 没有自动使用 Skill 怎么办？

**A:** 先检查 Skill 是否放在正确目录，再检查 `description` 是否明确写出触发场景。如果平台不支持自动发现，需要通过配置文件显式挂载。

### Q4: 有了 Skill 还需要 MCP 或工具吗？

**A:** 需要。Skill 说明“怎么做”，工具或 MCP 提供“能执行什么”。Skill 不能替代真实工具接口。

### Q5: 有了 Skill 还需要 Harness 吗？

**A:** 需要。Skill 只是能力说明和流程约束，不是安全边界。涉及文件、网络、数据库或系统命令时，仍需要 Harness 或平台权限控制。

### Q6: 为什么有 Skill 后 Token 消耗可能反而增加？

**A:** 因为 Skill 被触发后也会加载内容。判断 Skill 是否值得使用，要同时看输出质量、返工次数、总对话轮次、可复现性和人工修改成本。

### Q7: AI 总结出来的 Skill 可以直接使用吗？

**A:** 不建议直接使用。AI 可能把一次性细节写成通用规则，也可能保留敏感信息。必须人工校订后再挂载。
