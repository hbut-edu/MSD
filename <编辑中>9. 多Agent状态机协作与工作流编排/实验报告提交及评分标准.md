## 二、实验报告目录

1. 实验目的
2. 实验环境
3. 状态对象设计
4. 程序员 Agent 实现
5. 审查员 Agent 实现
6. 测试员 Agent 实现
7. 状态机调度逻辑
8. 工作流运行日志分析
9. 实验总结与心得体会

## 三、提交材料

1. 实验报告文档（Markdown、Word 或 PDF）
2. 源码文件：
   - `state.py`
   - `agents.py`
   - `workflow.py`
   - `main.py`
3. 工作流运行截图
4. 状态转移日志 JSON 或终端输出
5. 至少 1 个扩展节点或扩展思路说明

## 四、评分标准

| 评分项 | 分值 | 评分要点 |
| --- | ---: | --- |
| 状态对象设计 | 15 | 字段完整，能表达草稿、反馈、测试结果和历史 |
| Agent 节点实现 | 25 | 程序员、审查员、测试员职责清晰 |
| 状态机调度器 | 25 | 能实现路由、有限重试和终止条件 |
| 日志与可观测性 | 15 | 能记录每一步节点、消息和状态变化 |
| 运行结果 | 10 | 能展示至少一次驳回与修正闭环 |
| 分析与反思 | 10 | 能说明多 Agent 协作的优势和复杂度 |

## 五、加分项

- 增加安全审查、文档生成或格式检查节点：+4 分
- 将日志保存为 JSON 文件并可视化：+4 分
- 使用真实大模型替换某个 Agent 节点：+5 分

## 六、注意事项

1. 不要自动执行不可控的模型生成代码。
2. 若使用 `exec`，必须说明其教学演示性质和安全风险。
3. 工作流必须设置最大重试次数。
4. 报告应包含失败流转，而不仅展示最终成功结果。

## 七、常见问题

### Q1: 只实现两个 Agent 可以吗？

核心要求建议至少包含程序员与审查员。加入测试员更能体现完整工作流。

### Q2: 如果第一轮就通过了怎么办？

可以故意设计一个初始错误版本，观察驳回与修正过程。
