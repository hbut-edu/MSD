Git 冲突解决可视化

模拟三方合并（3-way Merge）中产生的冲突并手动仲裁

场景设定

你在 main 分支修改了函数名，而同事在 feature 分支修改了同一行的参数。

当前分支 (HEAD/main)

function sayHello(name)

引入分支 (feature)

function greet(user)

冲突识别

当 Git 无法自动合并时，会向文件注入冲突标记：

  • \( \ll\ll\ll\ll\ll\ll\ll \text{ HEAD} \)：当前更改开始
  • \( \text{=======} \)：内容分隔符
  • \( \gg\gg\gg\gg\gg\gg\gg \text{ branch} \)：引入更改结束

index.js — Conflicted
// 共同祖先部分
import { log } from './utils';
<<<<<<< HEAD (当前更改)
function sayHello(name) {
=======
function greet(user) {
>>>>>>> feature-branch (引入的更改)
console.log("Welcome!");
}

✓ 冲突标记已清除，代码已就绪

Git 状态： 检测到合并冲突。编辑文件以解决冲突，然后提交结果。

什么是三方合并？

Git 比较 \( \text{Base} \)、\( \text{HEAD} \) 和 \( \text{Incoming} \) 三个版本。如果 HEAD 和 Incoming 对同一位置进行了不同修改，Git 就无法确定保留哪个，从而产生冲突。

手动仲裁逻辑

用户需要删除标记并保留最终代码：
\( \text{Conflicted State} + \text{Manual Edit} \rightarrow \text{Resolved State} \)

提交闭环

冲突解决后，必须执行提交操作。此时文件的状态从 "Unmerged" 变为 "Staged"，最后进入版本历史库，完成合并生命周期。

合并成功！

你已经成功识别了冲突标记，进行了手动仲裁，并完成了提交闭环。
