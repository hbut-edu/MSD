## 四、评分标准

| 评分项 | 分值 | 评分要点 |
| --- | ---: | --- |
| 工作流封装 | 15 | 能将本地 AI 能力封装为函数并返回结构化结果 |
| FastAPI 接口 | 20 | 有健康检查和核心业务接口，请求/响应结构清晰 |
| 鉴权与错误处理 | 20 | 能拒绝无 Token 请求，并返回可理解错误 |
| 前端调用 | 15 | 能通过页面调用 API 并展示结果 |
| Skill/API 契约 | 10 | `SKILL.md` 能说明能力范围、接口约定和失败处理 |
| Docker 交付 | 10 | 能编写 Dockerfile 并说明镜像运行方式 |
| 测试与分析 | 10 | 能展示成功、失败、异常场景 |
| 报告表达 | 5 | 能说明从脚本到服务的工程价值 |

## 五、加分项

- 使用环境变量读取 Token：+3 分
- 接入前序 RAG 或 Agent 工作流：+5 分
- 增加接口日志和请求耗时统计：+4 分
- 增加更完整的前端交互状态：+3 分

## 六、注意事项

1. 不要提交真实 Token、API Key 或 `.env` 文件。
2. 本地服务不要暴露到公网。
3. Docker 镜像中不要复制无关隐私文件。
4. 报告应包含失败调用截图，体现鉴权有效。

## 七、常见问题

### Q1: 没有 Docker 可以完成实验吗？

可以。Docker 部分可作为增强项，本地 FastAPI 服务是核心。

### Q2: 前端页面必须很漂亮吗？

不需要。本实验重点是接口调用链路，页面清晰可用即可。
