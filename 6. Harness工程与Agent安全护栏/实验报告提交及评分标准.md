## 三、报告内容要求

### 3.1 实验目的与系统定位

报告应说明本实验为什么位于“Agent 能调用工具”和“Agent 能安全调用工具”之间，并解释以下关系：

- Skill 负责说明能力。
- Agent 根据用户意图生成工具请求。
- Harness 在工具执行前做代码级校验。
- Tool 只执行被放行的原子操作。
- 审计日志与失败反馈用于复盘和修正。

### 3.2 Skill、Tool 与 Harness 职责边界

报告应说明：

- Skill 为什么是能力说明，而不是权限系统。
- Tool 为什么应保持原子化，不直接处理 Agent 请求。
- Harness 为什么必须作为统一执行入口。
- Prompt 软约束与 Harness 硬约束有什么区别。

### 3.3 工具请求协议与威胁模型

报告应解释 `tool`、`path`、`content` 三个字段的含义，并列出本实验中的主要风险类型，例如：

- 路径穿越：`../blocked/secret.txt`
- 未授权工具：`delete_file`
- 危险文件类型：`.sh`
- 参数类型错误：`content` 不是字符串
- 写入内容过长
- 读取类工具夹带 `content`

报告应说明 Harness 为什么应采用默认拒绝（fail closed）的策略。

### 3.4 实现说明

报告应结合代码说明以下文件的职责：

- `policy.py`：集中定义工作区、日志路径、工具白名单、扩展名白名单和写入长度限制。
- `tools.py`：实现 `read_text`、`write_text`、`append_text`、`list_files` 等原子工具。
- `harness.py`：实现 `resolve_workspace_path`、`validate_request`、`run_tool`、`audit`、`build_feedback`。
- `main.py`：模拟 Agent 请求，观察成功、拦截和反馈。
- `test_harness.py`：用自动化测试验证护栏是否有效。

### 3.5 测试与结果分析

报告应展示并解释：

- 合法读取或写入请求为什么能成功。
- 至少 4 类失败请求为什么被拦截。
- `pytest -q` 的运行结果。
- `audit.jsonl` 中成功与失败记录的字段含义。
- `build_feedback` 如何把结构化结果转换为可回注给 Agent 的反馈。

## 六、注意事项

1. 不要在实验中读取真实隐私文件，`blocked/secret.txt` 只用于模拟越权测试。
2. 不要开放真实删除、格式化、系统命令执行等危险能力。
3. 日志示例应使用模拟数据，不要包含真实 Token、密码、Cookie 或个人隐私信息。
4. 报告应说明每类拦截的原因，而不仅展示截图或终端输出。
5. 若使用真实大模型扩展实验，模型输出必须先转换为工具请求，再交给 Harness 校验，不能直接执行。
6. 若同学对 Skill 概念不熟悉，可先将其理解为“面向 Agent 的能力说明书”，重点关注它与 Harness 的职责差异。

## 七、常见问题

### Q1: 没有使用真实大模型会影响成绩吗？

不会。本实验重点是 Harness 的工程实现，模拟 Agent 请求即可完成核心目标。真实模型只是工具请求的来源之一，不应改变 Harness 校验层。

### Q2: pytest 不是必须的吗？

可以用普通 `assert` 或终端输出验证，但自动化测试更能体现工程能力。评分中“自动化安全测试”要求建议使用 `pytest -q` 展示结果。

### Q3: 为什么要提交 `safe_file_skill/SKILL.md`？

因为本实验要求同学理解“能力说明”和“执行边界”的区别。`SKILL.md` 用于描述能力范围和禁止行为，Harness 才负责真正拦截不合规请求。

### Q4: `logs/audit.jsonl` 需要提交完整文件吗？

不要求提交很长的完整日志。报告中展示或附上示例内容即可，但示例应至少包含一次成功请求和一次失败请求。

### Q5: 如果测试失败怎么办？

先检查是否在 `harness_guardrail_lab` 目录下运行测试，再确认 `workspace/note.md` 是否存在。若路径和文件都正确，再检查 `validate_request`、`resolve_workspace_path` 和工具白名单逻辑。
