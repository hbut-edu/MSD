## 四、评分标准

| 评分项 | 分值 | 评分要点 |
| --- | ---: | --- |
| 评估样本集 | 20 | 覆盖正常、资料外、攻击、隐私等类型 |
| 目标系统实现 | 15 | 能对不同输入返回可评估输出 |
| Evaluator 设计 | 25 | 能检查 JSON、关键词、拒答、脱敏等指标 |
| 批量评估报告 | 15 | 能输出通过率、分类型统计和失败详情 |
| 安全与隐私测试 | 15 | 能展示注入拦截和敏感信息脱敏 |
| 伦理反思 | 10 | 能结合工程责任讨论 AI 系统边界 |

## 五、加分项

- 将评估框架接入前序 RAG、Harness 或 API 项目：+5 分
- 增加 LLM-as-a-Judge 语义评估：+5 分
- 将报告导出为 CSV 或 HTML 仪表盘：+4 分
- 增加更多安全攻击样本并分类统计：+4 分

## 六、注意事项

1. 不得使用真实隐私数据进行测试。
2. 安全攻击样本只能用于本地教学环境。
3. 报告中不得展示真实 API Key、Token 或账号密码。
4. AI 辅助生成的伦理反思必须由本人核验和修改。
5. 报告应包含失败样本分析，不能只展示通过率。

## 七、常见问题

### Q1: 评估集需要多少条样本？

建议至少 8 条，覆盖正常、边界、拒答、安全攻击和隐私脱敏。样本质量比数量更重要。

### Q2: Evaluator 写得简单会不会不够？

规则 Evaluator 可以很简单，但指标必须明确、可解释、可重复运行。

### Q3: 伦理反思应该写什么？

重点讨论工程师如何通过权限、日志、评估、脱敏和人工复核降低 AI 系统风险。
